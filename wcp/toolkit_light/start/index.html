Wirecard Checkout Toolkit light

Wirecard Checkout Toolkit light

What is the Wirecard Checkout Toolkit light?

Within the usage of the Wirecard Checkout Toolkit light you have the possibility to use commands which enables various operations regarding the orders of your consumer. These operations are called by a server-to-server request from your web server to the Wirecard Checkout Platform. So you have the possibility either to call these operations automatically from the back-end of your web shop or manually from your web interface you have implemented for administration purposes.

Which operations are available?

The Wirecard Checkout Toolkit light offers you the following operations:

  • #approveReversal for cancelling an approval

  • #deposit for converting an approval to a posting

  • #depositReversal for cancelling a posting

  • #getOrderDetails for returning all order’s detail including corresponding payments and credit notes

  • #recurPayment for creating a new order based on an existing order

  • #refund for creating a credit note for an order

  • #refundReversal for cancelling a credit note

Please be aware that the availability of an operation depends on:

  • the payment type

  • the financial institution

  • the current state of the payment/order

In any case you always have the possibility to get all current available operations of a specific order by using the operation "getOrderDetails".

How is an operation started?

To start an operation you have to send a server-to-server request from your web server to the Wirecard Checkout Platform to the following URL:

https://secure.wirecard-cee.com/qpay/toolkit.php

Additionally you have to set specific POST-parameters depending on the operation you want to use.

Please be aware that it is sometimes necessary to enable server-to-server requests within the configuration of your web server. This issue arises typically on provider managed web servers with PHP.

For a proper request you have to set a correct HTTP header. Therefore you need to set the following HTTP header elements within your request:

HTTP header parameter Description

Host

Domain name of server. Has to be set to the following value: secure.wirecard-cee.com

User-Agent

User agent string of client.

Content-Type

MIME type of the body. Has to be set to the following value: application/x-form-urlencoded

Content-Length

Length of body in bytes.

Connection

Type of connection. Has to be set to the following value: close

Please be aware that an incorrect setting of the header parameters results to an HTTP 403 error message of the Wirecard Checkout Platform.

How is the fingerprint computed?

You can find an introduction to fingerprint and secret at Secret and fingerprint.

The fingerprint is computed by concatenating all request parameters and your secret without any dividers in between. If you do not use optional parameters you have to ignore them in your fingerprint string.

Please be aware that the concatenation of the request parameters and the secret has to be done in the order as defined within the detailed description of each operation.

After concatenating all values to a single string you use a MD5 hash and the result is the fingerprint which you add as a request parameter to the server-to-server call.

The Wirecard Checkout Platform knows also your secret and is able to check if the received parameters are not manipulated by a 3rd party. Therefore it is essential that only you and Wirecard knows your secret!

Which request parameters are required for all operations?

To start an operation you have to set all required parameters to their corresponding values. If one or more of these required parameters are missing you will get an error message.

How are the values returned?

After you send the request as a server-to-server request from your web server to the Wirecard Checkout Platform you will get the result of the operation as key-values pairs returned in the content of the response.

Which response parameters are returned?

Detailed description of each operation

approveReversal

This operation cancels an approval. This can only be done when the approval has not yet been debited and is supported by the financial service provider.

The required order of the parameter values when computing the fingerprint is: customerId, shopId, toolkitPassword, secret, command, language and orderNumber

Additional required request parameters
Parameter Data type Short description

orderNumber

numerical with a fixed length of 9

order number of payment
Additional response parameters

Within this operation no additional response parameters are returned.

deposit

This operation causes an approval to be converted into a posting. This operation is executed by the merchant if an order of a consumer has been accepted. Additionally a daily-closing is automatically opened when there is no one already open for this payment type and currency.

This operation can be reversed with the operation depositReversal when the corresponding daily-closing has not yet been done and the financial services provider and the payment type supports this operation.

The required order of the parameter values when computing the fingerprint is: customerId, shopId, toolkitPassword, secret, command, language, orderNumber, amount and currency

depositReversal
